# SP-4: Transactional Outbox (Spring Modulith → Kafka) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`).

**Goal:** Add a reusable `acme-outbox` starter that gives services a transactional outbox via Spring Modulith's Event Publication Registry with Kafka externalization, and prove it in `demo-service`: creating an order publishes an `OrderCreated` domain event inside the command transaction, Modulith persists it (outbox row) and externalizes it to a Kafka topic after commit — verified end-to-end against a real Redpanda broker.

**Architecture:** `acme-outbox` is a thin starter bundling `spring-modulith-starter-jpa` (the JPA-backed event publication registry = the transactional outbox), `spring-modulith-events-jackson` (JSON serialization), and `spring-modulith-events-kafka` (externalization). A domain event annotated `@Externalized("topic::key")` and published via `ApplicationEventPublisher` within a transaction is written to the `event_publication` table in that same transaction (at-least-once), then published to Kafka after commit and the row marked complete. The demo publishes `OrderCreated` from the `CreateOrderCommand` handler (already transactional via SP-3's `StronglyConsistent` gating). Verification uses a Redpanda Testcontainer (`@ServiceConnection`) and a plain Kafka consumer that asserts the externalized message arrives. JSON now; Avro + Schema Registry + the consuming/inbox side are deferred to SP-4b.

**Tech Stack:** Java 21, Spring Boot 3.5.6, Spring Modulith 1.4.6 (events JPA + Jackson + Kafka), Spring Kafka, Testcontainers Redpanda (`redpandadata/redpanda:v24.2.7`, locally cached), JUnit 5 + AssertJ + Awaitility.

> Spec: `docs/superpowers/specs/2026-06-17-acme-boilerplate-design.md` §5 (acme-outbox), §7 (effectively-once = outbox + inbox; outbox is at-least-once). Builds on SP-0..3.
> **Decomposition note:** SP-4 covers the OUTBOX (publish) side with JSON. SP-4b will add the consumer + inbox dedup and switch the wire format to Avro + Confluent Schema Registry (requires the Confluent Maven repo + Avro codegen).

```java
 * Domain event emitted when an order is created. Externalized to the Kafka topic "orders"
```

- [ ] **Step 3:** Modify `CreateOrderCommandHandler` to publish the event after saving. Replace its contents:
```java
package com.acme.demo;

import an.awesome.pipelinr.Command;
import org.springframework.context.ApplicationEventPublisher;
import org.springframework.stereotype.Component;

@Component
public class CreateOrderCommandHandler implements Command.Handler<CreateOrderCommand, Long> {

    private final OrderRepository orders;
    private final ApplicationEventPublisher events;

    public CreateOrderCommandHandler(OrderRepository orders, ApplicationEventPublisher events) {
        this.orders = orders;
        this.events = events;
    }

    @Override
    public Long handle(CreateOrderCommand command) {
        Order saved = orders.save(new Order(command.sku(), command.quantity()));
        // Published inside the command's transaction (StronglyConsistent): Modulith writes the
        // event_publication row in the same tx, then externalizes to Kafka after commit.
        events.publishEvent(new OrderCreated(saved.getId(), saved.getSku(), saved.getQuantity()));
        return saved.getId();
    }
}
```

- [ ] **Step 4:** Modify `examples/demo-service/src/main/resources/application.yaml` — add a `modulith` block under `spring:` (keep all existing keys). Add these keys under the existing `spring:` mapping:
```yaml
  modulith:
```

## Task 5: demo — outbox externalization integration test (the proof)

**Files:** Create `OutboxExternalizationIT.java`.

- [ ] **Step 1:** Create `examples/demo-service/src/test/java/com/acme/demo/OutboxExternalizationIT.java`:
```java
package com.acme.demo;

import static org.assertj.core.api.Assertions.assertThat;

import an.awesome.pipelinr.Pipeline;
import com.acme.test.PostgresTestcontainersConfiguration;
import com.acme.test.RedpandaTestcontainersConfiguration;
import java.time.Duration;
import java.util.HashMap;
import java.util.List;
import java.util.Map;
import org.apache.kafka.clients.consumer.Consumer;
import org.apache.kafka.clients.consumer.ConsumerConfig;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.apache.kafka.clients.consumer.ConsumerRecords;
import org.apache.kafka.common.serialization.StringDeserializer;
import org.awaitility.Awaitility;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.context.annotation.Import;
import org.springframework.kafka.core.KafkaAdmin;

@SpringBootTest
@Import({PostgresTestcontainersConfiguration.class, RedpandaTestcontainersConfiguration.class})
class OutboxExternalizationIT {

    @Autowired
    Pipeline pipeline;

    @Autowired
    KafkaAdmin kafkaAdmin;

    @Test
    void creatingAnOrderExternalizesOrderCreatedToKafka() {
        Long id = pipeline.send(new CreateOrderCommand("SKU-OUTBOX", 7));
        assertThat(id).isNotNull();

        String bootstrap = kafkaAdmin.getConfigurationProperties()
                .get("bootstrap.servers").toString();

        try (Consumer<String, String> consumer = newConsumer(bootstrap)) {
            consumer.subscribe(List.of("orders"));
            // Modulith externalizes after the command transaction commits; poll until it arrives.
            Awaitility.await().atMost(Duration.ofSeconds(20)).untilAsserted(() -> {
                ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));
                boolean found = false;
                for (ConsumerRecord<String, String> r : records) {
                    if (r.value() != null && r.value().contains("\"orderId\":" + id)
                            && r.value().contains("SKU-OUTBOX")) {
                        found = true;
                    }
                }
                assertThat(found).as("OrderCreated for id %s on topic 'orders'", id).isTrue();
            });
        }
    }

    private static Consumer<String, String> newConsumer(String bootstrap) {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrap);
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "outbox-it");
        props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        return new org.apache.kafka.clients.consumer.KafkaConsumer<>(props);
    }
}
```
> Proof: dispatching `CreateOrderCommand` (StronglyConsistent → runs in a tx) publishes `OrderCreated`; Modulith stores it in `event_publication` within that tx and externalizes to the `orders` topic after commit. A vanilla Kafka consumer reading from `earliest` finds the JSON message containing the order id and sku — the full transactional-outbox → Kafka path, end-to-end on a real broker.

```markdown
# Transactional outbox via Spring Modulith Event Publication Registry + Kafka externalization

## Context and Problem Statement

Event choreography needs reliable publish: a domain event must be persisted atomically with the
state change and delivered to Kafka, without dual-write loss and without XA.

## Decision Outcome

- Use Spring Modulith's JPA-backed Event Publication Registry as the transactional outbox: an
  `@Externalized` domain event published via `ApplicationEventPublisher` inside the business
  transaction is written to `event_publication` in that same transaction (at-least-once), then
  published to Kafka after commit and marked complete.
- The `acme-outbox` starter bundles `spring-modulith-starter-jpa` + `-events-jackson` (JSON) +
  `-events-kafka`. Modulith self-configures; the app declares the event + topic via `@Externalized`.
- Effectively-once end-to-end requires pairing this at-least-once outbox with an idempotent inbox
  on the consumer side — deferred to SP-4b (which also switches the wire format to Avro + Confluent
  Schema Registry).
- On Postgres (tested) Modulith auto-initializes its schema; on Oracle/prod the table is created via
  Flyway and auto-init disabled (per spec §5 / ADR 0002 rationale).
- Spring Modulith pinned to 1.4.6 (the 1.4.x line targets Spring Boot 3.5; 2.x targets Boot 4).

Full detail: `docs/superpowers/specs/2026-06-17-acme-boilerplate-design.md` §5 (acme-outbox).
```
